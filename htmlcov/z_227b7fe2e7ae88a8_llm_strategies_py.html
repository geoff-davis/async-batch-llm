90class LLMCallStrategy(ABC, Generic[TOutput]): 

91 """ 

92 Abstract base class for LLM call strategies. 

93 

94 A strategy encapsulates how LLM calls are made, including: 

95 - Resource initialization (caches, clients) 

96 - Call execution with retries 

97 - Resource cleanup 

98 

99 The framework calls: 

100 1. prepare() once per unique strategy instance before its first execution 

101 2. execute() for each attempt (including retries) 

102 3. cleanup() once per prepared strategy when the processor exits or shuts down 

103 """ 

104 

105 async def prepare(self) -> None: 

106 """ 

107 Initialize resources before making any LLM calls. 

108 

109 Called once per unique strategy instance before the first work item using 

110 that instance executes. Use this to set up shared caches, clients, etc. 

111 Per-item retry state belongs in execute()/on_error() via RetryState. 

112 

113 Default: no-op 

114 """ 

115 pass

117 @abstractmethod 

118 async def execute( 

119 self, prompt: str, attempt: int, timeout: float, state: "RetryState | None" = None 

120 ) -> tuple[TOutput, TokenUsage] | tuple[TOutput, TokenUsage, dict[str, Any] | None]: 

121 """ 

122 Execute an LLM call for the given attempt. 

123 

124 Args: 

125 prompt: The prompt to send to the LLM 

126 attempt: Which *logical* retry attempt this is (1, 2, 3, ...). 

127 **Guarantee:** rate-limit errors do NOT advance this number. If a 

128 call is throttled (429 / coordinated cooldown), the framework 

129 retries the *same* ``attempt`` after the cooldown rather than 

130 burning an attempt. So a strategy that escalates models on higher 

131 attempts (e.g. attempt >= 2 → smarter model) escalates because the 

132 *output* was bad over ``max_attempts`` tries, never merely because 

133 the endpoint was busy. Rate-limit retries are bounded separately 

134 by ``RetryConfig.max_rate_limit_retries``. 

135 timeout: Maximum time to wait for response (seconds) 

136 state: Optional retry state that persists across attempts (v0.3.0) 

137 

138 Returns: 

139 Either a 2-tuple ``(output, token_usage)`` or a 3-tuple 

140 ``(output, token_usage, metadata)``. ``token_usage`` is a 

141 TokenUsage dict with optional keys ``input_tokens``, 

142 ``output_tokens``, ``total_tokens``, ``cached_input_tokens``. 

143 ``metadata`` (v0.10.0) is a provider-specific dict forwarded into 

144 ``WorkItemResult.metadata`` — typically ``finish_reason``, 

145 ``model``, ``provider`` (OpenRouter), ``safety_ratings`` (Gemini); 

146 pass ``None`` if you have nothing to surface. The 2-tuple shape is 

147 supported for backward compatibility but will be removed in a 

148 future release; built-in strategies all return the 3-tuple shape. 

149 

150 Raises: 

151 Exception: Any exception propagated to trigger a retry (if 

152 retryable) or a permanent failure. 

153 

154 Note (v0.3.0): 

155 The state parameter allows strategies to maintain state across retry 

156 attempts for multi-stage retry patterns. See RetryState documentation 

157 for examples. 

158 """ 

159 pass
